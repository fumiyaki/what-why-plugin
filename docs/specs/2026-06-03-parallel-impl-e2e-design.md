# /ww-flow 並列実装・E2E ループ拡張 設計書

- 日付: 2026-06-03
- ステータス: 設計確定（実装計画 未作成）
- 対象: `what-why` プラグインの `/ww-flow` コマンド（`fumiyaki/what-why-plugin`）

## 1. 背景と目的

現状の `/ww-flow` は Phase A〜D（grill → 抽出 → 単一セッションでの逐次実装 → tmp 掃除）で、
メインセッション1つが順番に実装する。これを次のように拡張する。

- **実装フェーズの並列化**: Issue を PR 単位に分割し、PR ごとにチームメンバーを spawn して
  並列実装する。各 PR はレビューを通してから完成させる。
- **結合テスト**: 全 PR 完了後に結合してテストする。
- **E2E ループ**: ローカル環境で Playwright による E2E を回し、失敗したら担当メンバーへ
  差し戻して green になるまで（上限つきで）繰り返す。

目的は「grill で詰めた理解を、複数メンバーで並列に実装し、レビュー・結合・E2E まで
メインセッションが司令塔として統括して完遂する」こと。

## 2. エージェントモデル（最重要の前提）

検証により、**子エージェントを spawn できるのはメインセッションのみ**で、spawn された
チームメンバーは `Agent` ツールを持たない（内部サブエージェントを作れない）ことが確定した。
この制約に全設計を従わせる。メインが司令塔として全エージェントを spawn する。

使うエージェントは親が誰かではなく「対話継続の要否」で2種類に分ける。

| 役割 | 種別 | 生成方法 | 寿命 |
|---|---|---|---|
| 実装 | チームメンバー | `Agent` + `team_name` + `name` | **永続**（結合/E2E の再修正に備え残す）|
| コードレビュー | チームメンバー（fresh, 仕様だけ渡す）| `Agent` + `team_name` | 使い捨て（レビュー後シャットダウン）|
| E2E シナリオ作成 | 内部サブエージェント | `Agent`（`team_name` なし）| 使い捨て |
| E2E 失敗の原因切り分け | 内部サブエージェント | `Agent`（`team_name` なし）| 使い捨て |
| E2E 実行 | チームメンバー | `Agent` + `team_name` | 使い捨て（E2E 完了後シャットダウン）|

- **チームメンバー**: `SendMessage` で対話継続でき、再修正依頼ができる。実装・レビュー・
  E2E 実行のように「差し戻し→再対応」が起きうる役割に使う。
- **内部サブエージェント**: 結果だけ返す一発仕事（シナリオ作成・原因切り分け）に使う。
- メンバーは子を産めないため、レビューはメインが fresh メンバーを spawn して行う。fresh
  なので実装の会話コンテキストが一切混入せず、「仕様だけ渡す pure なレビュー」になる。

## 3. PR 粒度と commit 粒度

3段の階層を持つ。

```
Sub-issue / PR グループ  ← メンバー単位（1メンバー = 1 PR）
  └ What                 ← commit 単位（1 What/Why = 1 commit）
      └ 実装
```

- **PR 単位**: 親 Issue に GitHub Sub-issue があれば、各 Sub-issue = 1 PR = 1 メンバー。
  Sub-issue が無ければ Phase B が What を中粒度の PR グループにまとめて PR 単位を決める。
- **commit 単位**: どの PR の中でも「1 What/Why = 1 commit」を厳守する（既存の `what-why`
  スキル＋`COMMIT-FORMAT.md` の鉄則を維持）。
- PR 間には依存関係がありうる。Phase B で依存グラフを構築し、独立 PR は並列・依存ありは
  直列で実装する。

## 4. 作業空間の分離

並列実装するメンバーは git worktree でメンバーごとに独立した作業ディレクトリ＋ブランチを
持つ（同一リポジトリの git index 競合を防ぐ）。`Agent` の `isolation: "worktree"` を使う。
worktree の掃除は最終 Phase（F）で行う。E2E の再修正まで作業空間を残す必要があるため、
途中では掃除しない。

## 5. Phase 構成

`/ww-flow` は次の6 Phase になる。Phase A は既存のまま。B 以降を拡張・新設する。

### Phase A — grill（既存）

`grill-with-docs` スキルで Issue を詰める。仕様理解＋ドメイン用語の `CONTEXT.md` 育成。
この Phase では実装に着手しない。

### Phase B — 抽出＋計画（拡張）

grill 完了後、成果を仕分けて実装計画を立てる。

1. `what-why` スキルを読み込む。
2. What/Why を抽出し、該当ドメイン台帳の「やること（進行中）」へ記録する（既存）。
3. How を `what-why/tmp/working-notes.md` へ書き出す（既存）。
4. **PR 単位を決める（新規）**: 親 Issue の Sub-issue を確認する。
   - Sub-issue があれば、各 Sub-issue を1 PR とする。
   - Sub-issue が無ければ、抽出した What を機能のまとまりで中粒度の PR グループにまとめる。
5. **PR 依存グラフを構築する（新規）**: PR 間の依存（先行 PR の成果に依存するか）を判定し、
   独立 PR と依存 PR を区別する。

### Phase C — 並列実装（新規）

依存グラフに従って実装メンバーを spawn し、PR を実装・レビュー・完成させる。

1. **メンバー spawn**: 独立した（依存元が無い／依存元が完了済みの）PR について、実装メンバーを
   `Agent` + `team_name` + `isolation: "worktree"` で spawn する。独立 PR は並列に spawn し、
   依存 PR は依存元の完了後に spawn する。
2. **実装**: 各メンバーは担当 PR を「1 What/Why = 1 commit」で実装する。commit 作法は
   `what-why` スキル／`COMMIT-FORMAT.md` に従う。
3. **レビュー**: メンバーが PR を実装し終えたら、メインが fresh のレビューメンバーを spawn する。
   レビューメンバーには **PR の仕様（担当 Sub-issue / PR グループの What/Why）と差分の git
   範囲だけ**を渡し、実装の会話コンテキストは渡さない。spec 準拠 → コード品質 の二段で見る。
4. **差し戻しループ**: レビューで問題が出たら、担当の実装メンバーへ `SendMessage` で修正を
   依頼する。レビュー再実行 → 通過するまで繰り返す。通過したらレビューメンバーをシャット
   ダウンする。
5. **実装メンバーは消さない**: PR 完成後も実装メンバーは残す（Phase D/E の再修正に備える）。
6. 全 PR が完成するまで 1〜5 を続ける。

### Phase D — 結合テスト（新規）

全 PR 完了後、成果を結合してテストする。

1. 各 PR を結合する（マージ順は依存グラフに従う）。
2. テスト（プロジェクトのテストコマンド）を実行する。
3. 問題が出たら、原因に対応する実装メンバーへ `SendMessage` で再修正を依頼し、再結合・
   再テストする。問題が解消するまで繰り返す。

### Phase E — E2E ループ（新規）

ローカル環境で Playwright による E2E を回す。メインは判断と統括に徹し、実作業は委譲する。

1. **シナリオ作成**: メインが内部サブエージェント（使い捨て）を spawn し、実装内容から
   E2E シナリオを考えて一覧化させる。結果（シナリオ一覧）をメインが受け取る。
2. **E2E 実行**: メインが E2E 実行メンバーを spawn し、シナリオ一覧を渡して Playwright MCP
   （`mcp__plugin_playwright__*`）でローカル環境に対して実行させる。ローカル dev サーバの
   起動が前提（対象プロジェクトの dev 起動コマンドで立てた URL に向ける）。
3. **失敗時の切り分け**: シナリオが失敗したら、メインが内部サブエージェント（使い捨て）を
   spawn し、失敗内容と各メンバーの担当 PR 範囲を渡して「どのメンバーの実装漏れか」を
   特定させる。結果をメインが受け取る。
4. **差し戻し**: メインが特定された実装メンバーへ `SendMessage` で修正を依頼する。修正完了後、
   E2E を再開する。
5. **リトライ上限**: リトライは**シナリオ単位で上限3回**。同一シナリオが3回差し戻しても
   green にならなければ、メインはユーザーに状況を報告して停止する（暴走防止の安全弁）。
6. 全シナリオが green になるまで（または上限超過まで）2〜5 を繰り返す。

### Phase F — クリーンアップ（拡張）

全工程完了後に後片付けする。

1. 全実装メンバー（および残存メンバー）を `SendMessage` の `shutdown_request` で
   シャットダウンする。
2. `git worktree remove` で各メンバーの worktree を除去し、マージ済みブランチを削除する。
3. `TeamDelete` でチームを削除する。
4. `what-why/tmp/` ディレクトリを削除する（永続記録は commit メッセージ＝git log に残る）。

## 6. 設計判断の記録

| 判断 | 理由 |
|---|---|
| 子を spawn するのはメインのみ | 検証で teammate は `Agent` ツールを持たない（子を産めない）と判明。メイン統括に一本化 |
| 実装/レビュー/E2E実行=メンバー、シナリオ作成/切り分け=サブエージェント | 対話継続（差し戻し）の要否で種別を分けた。一発仕事はサブエージェント |
| レビューは fresh メンバーに仕様だけ渡す | 実装会話の混入を構造的に防ぎ、pure なレビューにする |
| PR 単位は Sub-issue 優先・無ければ中粒度グループ | Sub-issue 前提だと汎用性が落ち、commit 粒度だと PR が荒立つため中間を取る |
| commit は 1 What/Why = 1 commit を厳守 | 既存 what-why の記録規律を並列化後も維持する |
| 並列実装は worktree 分離 | 同一リポジトリの git index 競合を防ぐ。`Agent` の isolation 機能を使う |
| E2E は Playwright MCP | シナリオ達成判定を安定して自動化でき、実ブラウザを奪い合わない |
| E2E リトライはシナリオ単位3回 | 失敗の「深さ」で打ち切る。多数シナリオが1回ずつ失敗（健全）は許容、同一シナリオの連続失敗で停止 |
| worktree 掃除は最終 Phase F | E2E の再修正まで作業空間を残す必要があるため途中で掃除しない |

- ローカル dev サーバの起動を誰が（どのメンバーが）担うか、対象プロジェクトごとの

  プロジェクト内ファイルからの推定）。
- レビューメンバーの二段レビュー（spec 準拠→コード品質）のプロンプトを既存の
  subagent-driven-development のテンプレートに合わせるか、独自に持つか。
- 並列実装の同時メンバー数の上限（リソース・レート制限）。
