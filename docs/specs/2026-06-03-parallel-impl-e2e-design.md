# /ww-flow 並列実装・E2E ループ拡張 設計書

- 日付: 2026-06-03
- ステータス: 設計確定（v2 — ワークフローレビューと実機検証を反映）
- 対象: `what-why` プラグインの `/ww-flow` コマンド（`fumiyaki/what-why-plugin`）

> **v2 改訂の経緯**: v1 をワークフローとしてレビューした結果、(1) 並列メンバーが共有台帳に
> 同時書き込みして壊れる、(2) 依存 PR の worktree ベースが `origin/main` 固定で依存元の
> コードが見えない、(3) マージの主体・タイミング未定義、という Critical を検出。加えて実機
> 検証で **`Agent` の `isolation: "worktree"` がチームメンバーには効かず、プライマリ作業
> ツリーで動く**ことが判明した。これらを受けて以下を確定した:
> - 台帳（WHAT-WHY.md / CONTEXT.md / tmp）の書き込みは**メインに一本化**（メンバーは commit のみ）
> - worktree は **メインが手動で `git worktree add`** して管理（`isolation` param に依存しない）
> - 依存 PR は**依存元ブランチをベースに** worktree を切る（スタック PR）
> - **マージは人間**。ワークフローは PR 作成まで。D/E は**ローカル統合ブランチ**で検証
> - Phase D の結合テスト失敗も**3回上限**、異常停止時のクリーンアップとメンバー**再 spawn
>   リカバリ手順**を新設

## 1. 背景と目的

現状の `/ww-flow` は Phase A〜D（grill → 抽出 → 単一セッションでの逐次実装 → tmp 掃除）で、
メインセッション1つが順番に実装する。これを次のように拡張する。

- **実装フェーズの並列化**: Issue を PR 単位に分割し、PR ごとにチームメンバーを spawn して
  並列実装する。各 PR はレビューを通してから完成させる。
- **結合テスト**: 全 PR 完了後に結合してテストする。
- **E2E ループ**: ローカル環境で Playwright による E2E を回し、失敗したら担当メンバーへ
  差し戻して green になるまで（上限つきで）繰り返す。

目的は「grill で詰めた理解を、複数メンバーで並列に実装し、レビュー・結合・E2E まで
メインセッションが司令塔として統括して完遂する」こと。

## 2. エージェントモデル（最重要の前提）

検証により、**子エージェントを spawn できるのはメインセッションのみ**で、spawn された
チームメンバーは `Agent` ツールを持たない（内部サブエージェントを作れない）ことが確定した。
この制約に全設計を従わせる。メインが司令塔として全エージェントを spawn する。

使うエージェントは親が誰かではなく「対話継続の要否」で2種類に分ける。

| 役割 | 種別 | 生成方法 | 寿命 |
|---|---|---|---|
| 実装 | チームメンバー | `Agent` + `team_name` + `name` | **永続**（結合/E2E の再修正に備え残す）|
| コードレビュー | チームメンバー（fresh, 仕様だけ渡す）| `Agent` + `team_name` | 使い捨て（レビュー後シャットダウン）|
| E2E シナリオ作成 | 内部サブエージェント | `Agent`（`team_name` なし）| 使い捨て |
| E2E 失敗の原因切り分け | 内部サブエージェント | `Agent`（`team_name` なし）| 使い捨て |
| E2E 実行 | チームメンバー | `Agent` + `team_name` | 使い捨て（E2E 完了後シャットダウン）|

- **チームメンバー**: `SendMessage` で対話継続でき、再修正依頼ができる。実装・レビュー・
  E2E 実行のように「差し戻し→再対応」が起きうる役割に使う。
- **内部サブエージェント**: 結果だけ返す一発仕事（シナリオ作成・原因切り分け）に使う。
- メンバーは子を産めないため、レビューはメインが fresh メンバーを spawn して行う。fresh
  なので実装の会話コンテキストが一切混入せず、「仕様だけ渡す pure なレビュー」になる。

### 共有資産の書き込みはメインに一本化（v2）

台帳ファイル群（`WHAT-WHY.md` 各ドメイン台帳・`CONTEXT.md` / `CONTEXT-MAP.md`・
`what-why/tmp/`）への**書き込みはメインセッションだけが行う**。実装メンバーは担当 PR の
**コード commit のみ**を行い、台帳には触れない。理由: 複数メンバーが同じ台帳ファイルを
同時編集するとロストアップデート・恒常コンフリクトが起きるため（v1 の Critical C1）。
メンバーは「実装した What/Why/How」を報告し、台帳への反映はメインが直列に行う。

## 3. PR 粒度と commit 粒度

3段の階層を持つ。

```
Sub-issue / PR グループ  ← メンバー単位（1メンバー = 1 PR）
  └ What                 ← commit 単位（1 What/Why = 1 commit）
      └ 実装
```

- **PR 単位**: 親 Issue に GitHub Sub-issue があれば、各 Sub-issue = 1 PR = 1 メンバー。
  Sub-issue が無ければ Phase B が What を中粒度の PR グループにまとめて PR 単位を決める。
- **commit 単位**: どの PR の中でも「1 What/Why = 1 commit」を厳守する（既存の `what-why`
  スキル＋`COMMIT-FORMAT.md` の鉄則を維持）。
- PR 間には依存関係がありうる。Phase B で依存グラフを構築し、独立 PR は並列・依存ありは
  直列で実装する。

## 4. 作業空間の分離（メインによる手動 worktree 管理）

並列実装するメンバーは、メンバーごとに独立した git worktree ＋ブランチを持つ（同一作業
ツリーの git index 競合を防ぐ）。

**`Agent` の `isolation: "worktree"` は使わない。** 実機検証の結果、チームメンバーに
`isolation: "worktree"` を指定しても分離されず、メンバーはプライマリ作業ツリーでそのまま
動き、現在ブランチに直接 commit してしまうことが判明した（リポジトリ汚染のリスク）。

代わりに**メインセッションが手動で worktree を作成・管理する**:

- メインが各 PR について `git worktree add <worktree-path> -b <pr-branch> <base-branch>`
  を実行する。
  - `<worktree-path>`: リポジトリ外の専用ディレクトリ（例 `<repo>/../.ww-flow/<pr-key>`）。
  - `<pr-branch>`: PR 用ブランチ名（例 `ww/<issue>/<pr-key>`）。
  - `<base-branch>`: **独立 PR は `main`（既定ブランチ）、依存 PR は依存元 PR のブランチ**
    （スタック PR。§5 参照）。
- メインは実装メンバーを spawn する際、プロンプトで「`<worktree-path>` で作業せよ」と
  明示的に作業ディレクトリを指示する。メンバーはそのパスの中だけで commit する。
- メインは各 PR の `<worktree-path>` / `<pr-branch>` / `<base-branch>` / 担当メンバー名 /
  PR 仕様 を**自身の記録として保持**する（リカバリと統合に使う。§5・§8 参照）。

worktree の掃除は最終 Phase（F）または異常停止時のクリーンアップで行う。E2E の再修正まで
作業空間を残す必要があるため、途中では掃除しない。

## 5. マージ・統合モデル

**マージの最終主体は人間**。ワークフローは各 PR を**作成するところまで**を行い、main への
マージは行わない。一方で Phase D（結合テスト）・E（E2E）は「統合された状態」のコードを
必要とするため、ワークフローは**push しないローカル統合**で検証する。

### ブランチ構造

- **独立 PR**（依存元なし）: `main` から派生。複数が並列に存在する。
- **依存 PR**（依存元あり）: 依存元 PR のブランチから派生（スタック PR）。一直線の依存
  チェーンなら、先端ブランチが依存先を全て含むため**先端＝そのチェーンの統合結果**になる。

### Phase D/E が検証する対象

- **依存チェーンが1本だけ**なら、その先端ブランチがそのまま統合済み。追加マージ不要。
- **独立 PR が複数ある**（依存フォレストが複数ルート）なら、どの単一ブランチにも全変更は
  含まれない。メインが**ローカル統合ブランチ `ww-flow/integration`** を作り、各依存フォレスト
  の leaf（先端）ブランチを全てマージして、統合状態を作る。
- Phase D はこの統合ブランチ（または単一チェーンの先端）に対してテストを実行する。
- Phase E はこの統合ブランチをローカル環境にデプロイして E2E を実行する。
- **`ww-flow/integration` は push しない**。あくまでローカル検証用。本物の main への取り込みは
  人間が GitHub 上で各 PR をマージして行う（依存順にマージ）。

## 6. Phase 構成

`/ww-flow` は次の6 Phase になる。Phase A は既存のまま。B 以降を拡張・新設する。

### Phase A — grill（既存）

`grill-with-docs` スキルで Issue を詰める。仕様理解＋ドメイン用語の `CONTEXT.md` 育成。
この Phase では実装に着手しない。

### Phase B — 抽出＋計画（拡張）

grill 完了後、成果を仕分けて実装計画を立てる。

1. `what-why` スキルを読み込む。
2. What/Why を抽出し、該当ドメイン台帳の「やること（進行中）」へ記録する（**メインが書く**）。
3. How を `what-why/tmp/working-notes.md` へ書き出す（**メインが書く**）。
4. **PR 単位を決める（新規）**: 親 Issue の Sub-issue を確認する。
   - Sub-issue があれば、各 Sub-issue を1 PR とする。
   - Sub-issue が無ければ、抽出した What を機能のまとまりで中粒度の PR グループにまとめる。
5. **PR 依存グラフを構築する（新規）**: PR 間の依存（先行 PR の成果に依存するか）を判定し、
   独立 PR と依存 PR を区別する。依存チェーンの構造（どの PR がどの PR の上に積まれるか）を
   メインが記録する。これが §4 の worktree ベース選択と §5 の統合に使われる。

### Phase C — 並列実装（新規）

依存グラフに従って実装メンバーを spawn し、PR を実装・レビュー・完成させる。

1. **worktree 準備（メイン）**: 着手する PR について、メインが `git worktree add` で作業
   ディレクトリ＋ブランチを作る（§4）。ベースは独立 PR なら `main`、依存 PR なら依存元
   PR のブランチ。
2. **メンバー spawn**: 独立した（依存元が無い／依存元が完了済みの）PR について、実装メンバーを
   `Agent` + `team_name` + `name` で spawn し、プロンプトで担当 worktree のパスを指示する。
   独立 PR は並列に spawn し、依存 PR は依存元 PR の完成後に spawn する。
3. **実装**: 各メンバーは担当 worktree 内で PR を「1 What/Why = 1 commit」で実装する。
   commit 作法は `what-why` スキル／`COMMIT-FORMAT.md` に従う。**メンバーはコード commit
   のみを行い、台帳ファイルには触れない**（台帳はメインが書く）。実装した What/Why/How は
   メインへ報告する。
4. **台帳反映（メイン）**: メンバーの報告を受け、メインが該当ドメイン台帳の「やること」→
   「やったこと」移動と How の tmp 反映を**直列に**行う。
5. **PR 作成**: メンバーが worktree のブランチを push し、PR を作成する（base は §5 の
   ブランチ構造に従う＝独立は main、依存は依存元ブランチ）。PR 番号と差分の git 範囲を報告する。
6. **レビュー**: メインが fresh のレビューメンバーを spawn する。レビューメンバーには **PR の
   仕様と差分の git 範囲だけ**を渡し、実装の会話コンテキストは渡さない。spec 準拠 →
   コード品質 の二段で見る。
7. **差し戻しループ**: レビューで問題が出たら、担当の実装メンバーへ `SendMessage` で修正を
   依頼する。レビュー再実行 → 通過するまで繰り返す。通過したらレビューメンバーをシャット
   ダウンする。
8. **実装メンバーは消さない**: PR 完成後も実装メンバーは残す（Phase D/E の再修正に備える）。
9. 全 PR が完成するまで 1〜8 を続ける。

### Phase D — 結合テスト（新規）

全 PR 完了後、成果を**ローカル統合**してテストする（§5）。main へはマージしない。

1. **ローカル統合（メイン）**: 単一の依存チェーンなら先端ブランチをそのまま使う。独立 PR が
   複数あるなら、メインが `ww-flow/integration` ブランチを作り各 leaf ブランチをマージする。
   マージでコンフリクトが出たら、関係する実装メンバーへ差し戻して解消する。
2. **テスト実行**: テストコマンドを推定（`package.json` の scripts / `Makefile` / `README`
   等から）し、ユーザーに提示して実行可否を確認してから走らせる。
3. **差し戻し**: 問題が出たら、原因に対応する実装メンバーへ `SendMessage` で再修正を依頼し、
   再統合・再テストする。
4. **リトライ上限**: 結合テストの失敗も**3回上限**（E2E と同じ安全弁）。3回直しても green に
   ならなければ、メインは状況をユーザーに報告して停止する。

### Phase E — E2E ループ（新規）

ローカル環境で Playwright による E2E を回す。メインは判断と統括に徹し、実作業は委譲する。

1. **シナリオ作成**: メインが内部サブエージェント（使い捨て）を spawn し、実装内容から
   E2E シナリオを考えて一覧化させる。結果（シナリオ一覧）をメインが受け取る。
2. **E2E 実行**: メインが E2E 実行メンバーを spawn し、シナリオ一覧を渡して Playwright MCP
   （`mcp__plugin_playwright_playwright__*`）でローカル環境に対して実行させる。E2E は
   **Phase D のローカル統合ブランチ**（`ww-flow/integration` または単一チェーンの先端）を
   チェックアウトした環境で行う。ローカル dev サーバの起動が前提（dev 起動コマンドを推定→
   ユーザー確認→起動し、その URL に向ける）。
3. **失敗時の切り分け**: シナリオが失敗したら、メインが内部サブエージェント（使い捨て）を
   spawn し、失敗内容と各メンバーの担当 PR 範囲を渡して「どのメンバーの実装漏れか」を
   特定させる。結果をメインが受け取る。
4. **差し戻し**: メインが特定された実装メンバーへ `SendMessage` で修正を依頼する。修正完了後、
   E2E を再開する。
5. **リトライ上限**: リトライは**シナリオ単位で上限3回**。同一シナリオが3回差し戻しても
   green にならなければ、メインはユーザーに状況を報告して停止する（暴走防止の安全弁）。
6. 全シナリオが green になるまで（または上限超過まで）2〜5 を繰り返す。

### Phase F — クリーンアップ（拡張）

全工程完了後にメインが後片付けする。

1. 全実装メンバー（および残存メンバー）を `SendMessage` の `shutdown_request` で
   シャットダウンする。
2. `git worktree remove` で各メンバーの worktree を除去し、PR 用ブランチのうち不要なものと
   ローカル統合ブランチ `ww-flow/integration` を削除する（PR 用ブランチは GitHub 上の PR が
   残るので、ローカルのみ削除）。
3. `TeamDelete` でチームを削除する。
4. **`what-why/tmp/` ディレクトリを削除する（メインが行う）**。永続記録は commit メッセージ
   （git log）に残る。**メンバーには tmp を削除させない**（早合点削除で他メンバーの How/ADR が
   失われる事故を防ぐ）。

## 7. リカバリと異常停止

### 実装メンバーが落ちた時のリカバリ

実装メンバーは永続だが、コンテキスト枯渇・タイムアウト等で落ちることがある。Phase D/E で
差し戻そうとした実装メンバーが不在の場合、メインは次の手順で復旧する。

1. 対象メンバーが完全に終了していることを確認する（team config の members から消えている）。
   shutdown 要求が処理中の同名メンバーへ再 spawn すると挙動が曖昧になるため、**完全終了を
   確認してから**再 spawn する。
2. 同じ名前で実装メンバーを再 spawn する（実機検証で同名再 spawn は可能と確認済み）。
3. 再 spawn したメンバーは fresh コンテキスト（前インスタンスの記憶を持たない）。よって
   メインが保持していた **担当 worktree のパス・ブランチ・PR 仕様・現在の commit 状態** を
   プロンプトで再提供し、作業を継続させる。worktree 自体はメンバーの寿命と独立してディスクに
   残るため、再アタッチできる。

### 異常停止時のクリーンアップ

Phase D/E のリトライ上限超過、または途中中断でワークフローが停止した場合、リソース
（メンバー・チーム・worktree・統合ブランチ）が残る。メインは停止をユーザーに報告する際、
次を明示する。

- 現在生存しているメンバーと、各 worktree のパス・ブランチ。
- ユーザーの選択肢: (a) 状態を保持したまま手動で調査・再開する / (b) Phase F のクリーンアップを
  実行してすべて破棄する。
- ユーザーが選ぶまで、メインは自動でリソースを破棄しない（調査の余地を残す）。

## 8. 設計判断の記録

| 判断 | 理由 |
|---|---|
| 子を spawn するのはメインのみ | 検証で teammate は `Agent` ツールを持たない（子を産めない）と判明。メイン統括に一本化 |
| 実装/レビュー/E2E実行=メンバー、シナリオ作成/切り分け=サブエージェント | 対話継続（差し戻し）の要否で種別を分けた。一発仕事はサブエージェント |
| レビューは fresh メンバーに仕様だけ渡す | 実装会話の混入を構造的に防ぎ、pure なレビューにする |
| 台帳・CONTEXT・tmp の書き込みはメインのみ（v2）| 複数メンバーの同時書き込みでロストアップデート/恒常コンフリクトが起きるため（C1）|
| worktree はメインが手動管理、`isolation` param は使わない（v2）| 実機検証で teammate に isolation が効かず、プライマリ作業ツリーを汚染すると判明 |
| 依存 PR は依存元ブランチをベースに切る（v2）| 依存元のコードが見える状態で実装するため（C2）。スタック PR を形成 |
| マージは人間、ワークフローは PR 作成まで（v2）| マージの最終判断を人間に残す。D/E は push しないローカル統合で検証 |
| D/E はローカル統合ブランチで検証（v2）| 独立 PR 複数を統合した状態をテストしつつ、main を汚さない |
| PR 単位は Sub-issue 優先・無ければ中粒度グループ | Sub-issue 前提だと汎用性が落ち、commit 粒度だと PR が荒立つため中間を取る |
| commit は 1 What/Why = 1 commit を厳守 | 既存 what-why の記録規律を並列化後も維持する |
| E2E は Playwright MCP | シナリオ達成判定を安定して自動化でき、実ブラウザを奪い合わない |
| 結合テスト・E2E リトライは3回上限（v2 で D も追加）| 失敗の「深さ」で打ち切る。無限ループを防ぐ安全弁 |
| tmp 削除はメインが Phase F で（v2）| メンバーの早合点削除で他メンバーの How/ADR が失われる事故を防ぐ |
| 同時メンバー数は無制限（v2）| 依存グラフ上独立な PR は全て同時 spawn する（ユーザー決定）|
| 異常停止時は自動破棄せずユーザーに選択を委ねる（v2）| 調査の余地を残す。リカバリは同名再 spawn＋情報再提供 |
| worktree 掃除は最終 Phase F | E2E の再修正まで作業空間を残す必要があるため途中で掃除しない |

## 9. 未確定・実装時に詰める点

- worktree の配置パス・ブランチ命名の具体規約（`<repo>/../.ww-flow/<pr-key>` /
  `ww/<issue>/<pr-key>` を仮置き。衝突回避とクリーンアップ容易性で確定する）。
- ローカル dev サーバの起動を誰が担うか（推定→ユーザー確認までは確定。起動の実行主体が
  メインか E2E メンバーか）。
- レビューメンバーの二段レビュー（spec 準拠→コード品質）のプロンプトを既存の
  subagent-driven-development のテンプレートに合わせるか、独自に持つか。
- ローカル統合でコンフリクトが頻発する場合の、差し戻し先（複数メンバーまたぎ）の決め方。
