# what-why-commit

実装作業を「1 commit = 1 What」で進め、各 What に Why を添えて記録するワークフロー。

## 適用条件

- リポジトリに `WHAT-WHY.md`（ルート直下または `what-why/` 配下）が存在する場合に適用する。
- 適用するのは**実装モード**（コード変更を伴う作業）のみ。調査・説明・レビューには適用しない。
- 未初期化のリポジトリでは `what-why-init` の実行を促す。

## 鉄則

- **What と Why の決定権は人間にある。** Claude は分解案・Why 候補を**提案**し、人間が**確定**する。Claude は採否を判断しない。
- 新しい指示が既存の What/Why と矛盾したら、解決せず質問に変換する:
  「以前の『○○』と今回の『△△』が矛盾します。どちらを今後の真実としますか？」
- How（実装方法）は WHAT-WHY.md に書かない。コード・PR・git log が正本。

## 着手前: What確定フェーズ

実装に入る前に必ず実行する。commit 時まで持ち越さない。

1. **分解** — ユーザーの依頼を commit サイズの What に分解する。1 What = 1 commit で完了する粒度。複合的・大きい依頼なら分解案を提示し、ユーザーに確認する。
2. **Why の確定** — 各 What について:
   - ユーザーが Why を述べていればそれを使う。
   - 述べていなければ、Why 候補を **3案**提示し、加えて「議論したい」を選択肢に含める。ユーザーは候補を選ぶ／議論する／自分でタイプする（AskUserQuestion を使うとよい）。
   - 自明な変更（typo 修正など）は Why に「なし」を許容する。**Why を捏造しない。**
3. **記録** — 確定した What/Why を `WHAT-WHY.md` の「やること（進行中）」に1行ずつ追記する。

実装中に新しい What が出たら、その場で同じ手順（分解 → Why → 記録）を通してから実装する。

## 実装

「やること」を1つずつ実装する。1 What 実装したら commit する。複数 What を1つの作業に混ぜない。

## commit

1. commit メッセージを次の構造で書く:
   ```
   <要約>

   What: <この commit で達成したこと（1つだけ）>
   Why:  <理由 / 自明なら「なし」>
   ```
2. `commit-msg` hook が構造を検証する。却下されたら構造を直して commit し直す（hook を回避しない）。
3. commit 後、対応する項目を `WHAT-WHY.md` の「やること」から「やったこと（完了）」へ Why ごと移す。

## WHAT-WHY.md の構造

```
## やること（進行中）
- <What> — なぜ: <Why>

## やらないこと
- <やらないこと> — なぜ: <Why>

## やったこと（完了）
<!-- 最終整理時の項目数: N -->
### <機能領域の見出し>
- <What> — なぜ: <Why>
```

- セクション順は「やること → やらないこと → やったこと」。
- 「やらないこと」に載せるのは「放っておくと誰かがやってしまう／繰り返し提案される」ものだけ。理由のない禁止は書かない。却下した案もここに書く。
- 項目に ID は振らない。参照は What 本文の引用で行う。
- 完了項目は捨てずに「やったこと」へ残す。git log は変更履歴であり、読めるアーカイブの代わりにはならない。

## やったこと のグルーピング

- 軸は **機能領域**（時系列ではない）。同じ領域が3件以上たまったら見出しを切る。
- 完了項目数が「最終整理時の項目数」+10 に達したら再整理する:
  1. 見出しを全部消す
  2. 全項目をフラットに戻す
  3. ランダムに並べ替える（旧グルーピングの順序バイアス除去）
  4. 機能領域で見出しを切り直す
  5. `<!-- 最終整理時の項目数: N -->` を現在数に更新
- 再整理のたび、採用した見出し集合を `WHAT-WHY-grouping-history.md` に記録する。連続2回で見出しが実質同じなら、グルーピングを固定してよい（固定の判断は人間）。
